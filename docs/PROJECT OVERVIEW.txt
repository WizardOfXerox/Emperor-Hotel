PROJECT OVERVIEW

Project name:
Emperor Hotel Reservation and Management System

Purpose:
This project is a web-based hotel reservation and management system built with Core PHP, OOP, MySQL, room XML + DOM, and Bootstrap-based responsive pages. Reservations are the core workflow, while the management modules support public hotel pages, user booking, admin walk-in operations, room inventory, check-in/check-out actions, guest history, payments, printable receipts, simulated transaction logs, users, and dashboard reporting.

TECH STACK

- Core PHP with OOP model classes
- MySQL database through PDO prepared statements
- Room XML import/export using DOMDocument
- Bootstrap 5 for responsive layout
- Local Chart.js 4.5.1 browser build for dashboard charts
- External CSS files under public/assets/css
- XAMPP-compatible local development structure

5. Room XML + DOM Integration

Status: Implemented for room records

- XML export is available in public/admin/rooms.php using rooms.php?export=xml.
- XML import is available through the Rooms admin page upload form.
- XML generation and parsing are implemented in app/models/Room.php.
- DOMDocument is used for export/import.
- XML support is intentionally limited to room records. Users, guests, reservations, and payments are managed through normal database CRUD pages.

Expected room XML fields:

</rooms>

7. AI Support System

Status: Implemented

- The support widget provides two scoped AI help modes: customer support (rooms, pricing, booking guidance, hotel info) and admin support (dashboard metrics, sales, occupancy, operational alerts).
- Local PHP dataset rules and direct database queries handle common questions first. The Gemini API is used only as a fallback for open-ended or conversational queries.
- The SupportAssistant model uses word-boundary greeting detection, phrase-coverage FAQ scoring, date/month range extraction (supporting specific month names, yesterday, last 30/90 days), and live database context injection.
- A step-by-step booking guide is built in for both guest self-service booking and admin walk-in reservation workflows.
- The support widget JavaScript normalizes Windows CRLF line endings and parses Markdown tables into structured HTML tables for clean in-chat rendering.
- The Gemini API payload uses camelCase systemInstruction and alternating user/model role turns with consecutive-role merging.
- Admin AI context includes live dashboard counters, monthly performance snapshots, room inventory status, operational check-out alerts, and range-specific revenue/occupancy data.
- Customer AI context includes hotel profile, available rooms with pricing, room catalog descriptions, and included perks.

CURRENT ROOM INVENTORY

The system uses three room types:

- Imperial Deluxe: 12 rooms, default seed price PHP 4,500.00
- Royal Executive: 12 rooms, default seed price PHP 7,500.00
- Emperor Presidential: 12 rooms, default seed price PHP 12,500.00

Room prices are stored in the database in rooms.price_per_night. They are not controlled by hardcoded public page values. Admins can update all rooms under one room type through the bulk price update form in public/admin/rooms.php.

Room type inclusions are stored as simple PHP catalog text. The default inclusions are:

- Imperial Deluxe: Complimentary breakfast set
- Royal Executive: Breakfast buffet plus priority Wi-Fi
- Emperor Presidential: Breakfast buffet, car shuttle, and late checkout

Room type inclusions are descriptive only. The reservation total is based on room price per night multiplied by the number of nights.

PAYMENT FLOW

- Reservations store the booking total.
- The Payments page is the source for manual payment entries and simulated transaction records.
- Transaction references are generated automatically when a payment is saved. Manual payments use a PAY prefix and simulated transactions use a SIM prefix.
- The Payments page cost tracker shows reservation total, confirmed paid amount, pending logs, balance due, and remaining payable amount.
- Active Pending and Confirmed transactions cannot exceed the reservation total.
- Simulated transactions are forced to Pending for admin review.
- Walk-in cash/card payments accepted at the counter can be marked Confirmed immediately.
- Pending simulated transactions should be reviewed by an admin and changed to Confirmed, Failed, or Refunded as needed.
- Confirmed, Failed, and Refunded transaction logs are locked after review so paid/history records are not accidentally edited.
- When confirmed payments cover the full reservation total, a Pending reservation is automatically promoted to Confirmed.
